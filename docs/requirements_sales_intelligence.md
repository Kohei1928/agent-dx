# 要件定義書: 企業キーパーソンリサーチ＆アウトリーチシステム

## 1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| プロジェクト名 | Sales Intelligence Platform |
| 作成日 | 2025年12月2日 |
| バージョン | 1.1 |

### 1.1 目的
営業活動における企業リサーチ・キーパーソン特定・アプローチを効率化・自動化するシステムを開発する。

### 1.2 背景
- 営業活動において、ターゲット企業のキーパーソン（決裁者・人事担当者等）を特定する作業に多大な時間がかかっている
- SNSアカウントの調査、採用状況の把握、連絡先の収集を手作業で行っている
- 営業メッセージの作成・送信・アプローチ履歴管理が属人化している
- これらの業務を自動化・一元管理することで、営業効率を大幅に向上させる

### 1.3 利用者
- 営業担当者：約10名（想定）
- ログイン認証：必要（Google OAuth想定）

### 1.4 コスト制約
- API費用月額上限: **10万円以内**
- 高精度優先（コストより精度を重視）

## 2.5 リサーチロジック詳細

### 2.5.1 キーパーソン検索クエリ

以下のクエリパターンで検索を実行:

| 検索クエリ | 対象役職 |
|-----------|---------|
| `"{会社名}" "代表"` | 代表取締役、代表者 |
| `"{会社名}" "代表取締役"` | 代表取締役 |
| `"{会社名}" "取締役"` | 取締役全般 |
| `"{会社名}" "執行役員"` | 執行役員 |
| `"{会社名}" "人事"` | 人事部長、人事担当者 |

### 2.5.2 役職の優先順位

全ての役職を検索するが、表示順序は以下:
1. 社長・代表取締役
2. 取締役
3. 執行役員
4. 人事部長
5. 人事担当者

### 2.5.3 確定/候補の判定基準

| 条件 | 判定 |
|------|------|
| 会社名 + 氏名（漢字）が一致 | **確定** |
| 会社名 + 氏名（ローマ字）が一致 | **確定** |
| 氏名のみ一致（会社名の記載なし） | **候補** |
| 同姓同名で複数アカウント発見 | **候補**（全て保存） |

**注意**: SNSアカウント名がローマ字の場合も考慮して照合

### 2.5.4 SNSアカウント候補

| 項目 | 設定値 |
|------|--------|
| 1人あたりの最大候補数 | **3件** |
| 対象プラットフォーム | LinkedIn, Facebook, X |
| 情報取得元の記録 | **必須**（URLも記録） |

### 2.5.5 リサーチ失敗時の対応

| 項目 | 設定値 |
|------|--------|
| リトライ回数 | **3回** |
| 情報なしの場合 | 「再リサーチ対象」として記録 |
| エラー通知 | 必要（理由を添えて画面・メール通知） |

## 3. ユースケース

### UC-001: 企業リストのインポート

```
【アクター】営業担当者
【事前条件】ログイン済み

【基本フロー】
1. 営業担当者が「企業インポート」画面を開く
2. CSVファイル（会社名、URL列を含む）をアップロード
   または手動で会社名・URLを入力
3. システムが重複チェックを実施
4. インポート確認画面で追加される企業一覧を表示
5. 「インポート実行」ボタンでDBに登録
6. バックグラウンドでリサーチジョブをキュー登録

【代替フロー】
- 3a. 重複企業がある場合、スキップ/上書きを選択可能
- 2b. 単一企業を手動入力で追加

【事後条件】
- 企業がDBに登録されている
- リサーチジョブがキューに登録されている
```

### UC-002: キーパーソン・SNSアカウント自動リサーチ

```
【アクター】システム（バックグラウンドジョブ）
【事前条件】企業がDBに登録されている

【基本フロー】
1. リサーチジョブが企業情報を取得
2. AI Web Research（GPT/SerpAPI等）でキーパーソン検索
   - 社長、取締役、部長、人事担当者等
3. 各キーパーソンのSNSアカウントを検索
   - Facebook、LinkedIn、X
4. 検索結果の確度を判定
   - 一致度が高い → 「確定」
   - 同姓同名・曖昧 → 「候補」（複数可）
5. 結果をDBに保存
6. リサーチ完了ステータスを更新

【代替フロー】
- 2a. 情報が見つからない場合は「情報なし」として記録
- 4a. 候補が複数ある場合はすべて保存し、人間が後で確定

【事後条件】
- キーパーソン情報がDBに保存されている
- SNSアカウント情報（確定/候補）が保存されている
```

### UC-003: 採用情報リサーチ

```
【アクター】システム（バックグラウンドジョブ）
【事前条件】企業がDBに登録されている

【基本フロー】
1. リサーチジョブが企業情報を取得
2. AI Web Researchで採用情報を検索
   - 出稿中の採用媒体（リクナビ、マイナビ、Green、Wantedly等）
   - 募集中のポジション
   - 担当者直通の電話番号・メールアドレス
   - 会社代表メールアドレス
3. 結果をDBに保存

【事後条件】
- 採用情報がDBに保存されている
```

### UC-004: 候補アカウントの確定（目視確認）

```
【アクター】営業担当者
【事前条件】SNSアカウント候補が登録されている

【基本フロー】
1. 営業担当者が「候補確認」画面を開く
2. 「候補」ステータスのSNSアカウント一覧を表示
3. 各候補について、リンク先のプロフィールを確認
4. 正しいアカウントを「確定」としてマーク
5. 誤りのアカウントを「除外」としてマーク

【事後条件】
- アカウントのステータスが更新されている
```

### UC-005: 営業メッセージテンプレートの登録

```
【アクター】営業担当者
【事前条件】ログイン済み

【基本フロー】
1. 営業担当者が「テンプレート管理」画面を開く
2. 「新規テンプレート」ボタンをクリック
3. 以下を入力:
   - テンプレート名
   - カテゴリ（初回コンタクト/フォローアップ/返信/お礼等）
   - メッセージ本文（変数埋め込み可）
   - 使用可能変数: {{人物名}}, {{会社名}}, {{役職}}, {{採用ポジション}}, {{自社名}}, {{署名}} 等
   - 生成AIへのプロンプト（文章生成の規則性）
4. 「保存」ボタンでテンプレート保存

【事後条件】
- テンプレートがDBに保存されている（全員共有）
```

**テンプレートカテゴリ:**
| カテゴリ | 説明 |
|---------|------|
| 初回コンタクト | 初めてのアプローチ |
| フォローアップ | 2回目以降のアプローチ |
| 返信 | 返信があった場合の対応 |
| お礼 | 面談後のお礼等 |

**AI生成時に渡す情報:**
- 会社情報（会社名、URL、業種、概要）
- 採用情報（出稿媒体、募集ポジション）
- キーパーソン情報（氏名、役職）
- 過去のアプローチ履歴

### UC-006: AIによる営業メッセージ生成

```
【アクター】営業担当者
【事前条件】
- キーパーソン情報が登録されている
- テンプレートが登録されている

【基本フロー】
1. 営業担当者が企業詳細画面でキーパーソンを選択
2. 「メッセージ生成」ボタンをクリック
3. テンプレートを選択
4. システムがテンプレート＋企業/人物情報を元にAI生成
5. 生成結果をプレビュー表示
6. 必要に応じてチャット形式で微調整
   例: 「もう少しカジュアルに」「採用ポジションを強調して」
7. 満足したら「確定」

【代替フロー】
- 6a. 直接テキストを編集することも可能
- 7a. 複数パターンを生成して比較することも可能

【事後条件】
- 生成メッセージが保存されている
```

### UC-007: SNS/メール送信

```
【アクター】営業担当者
【事前条件】メッセージが生成/確定されている

【基本フロー - API連携可能な場合】
1. 営業担当者が送信チャネル（LinkedIn, Facebook, X, メール）を選択
2. 「送信」ボタンをクリック
3. システムがAPIで送信を実行
4. 送信結果をアプローチ履歴に記録

【基本フロー - 半自動の場合】
1. 営業担当者が送信チャネルを選択
2. 「コピー＆送信画面を開く」ボタンをクリック
3. メッセージがクリップボードにコピーされる
4. 新タブで該当SNSの送信画面が開く
5. 営業担当者が手動で貼り付け・送信
6. 送信完了後「送信済みにする」ボタンをクリック
7. アプローチ履歴に記録

【事後条件】
- アプローチ履歴が記録されている
```

### UC-008: アプローチ状況の管理

```
【アクター】営業担当者
【事前条件】アプローチ履歴が記録されている

【基本フロー】
1. 営業担当者がキーパーソン詳細画面を開く
2. アプローチ履歴一覧を確認
   - いつ、どのチャネルで、どんなメッセージを送ったか
3. 結果ステータスを更新
   - 未着手 → 送信済み → 開封 → 返信あり → 面談設定 → 商談中 → 成約/失注
4. 必要に応じてメモを追加

【事後条件】
- ステータスが更新されている
```


### 7.4 SNS連携の現実的なアプローチ

| SNS | 方式 | 詳細 |
|-----|------|------|
| **LinkedIn** | Sales Navigator + 半自動 | メッセージ生成後、Sales Navigator画面で手動送信 |
| **Facebook** | 半自動 | コピー → Messenger画面を開く → 手動送信 |
| **X** | 半自動 | コピー → DM画面を開く → 手動送信 |
| **メール** | **自動送信** | Resend APIで自動送信可能 |

**半自動送信フロー:**
1. システムでメッセージを生成・確定
2. 「送信」ボタンクリック
3. メッセージがクリップボードにコピーされる
4. SNSの送信画面（DM画面）が新タブで開く
5. ユーザーが手動で貼り付け・送信
6. 「送信完了」ボタンでアプローチ履歴を記録

**LinkedIn自動化オプション（オプション）:**
- Phantombuster（$69/月〜）を使えば自動化可能
- ただしアカウント凍結リスクあり（自己責任）
